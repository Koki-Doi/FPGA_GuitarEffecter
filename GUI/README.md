# AudioLab HDMI GUI renderer

`GUI/` is the current Python-side renderer and control bridge used by the
AudioLab HDMI GUI work. It is **not** a standalone PYNQ base-overlay video
application.

Current live HDMI output uses:

- `AudioLabOverlay()` loaded exactly once
- `audio_lab_pynq.hdmi_backend.AudioLabHdmiBackend`
- the integrated HDMI path in `audio_lab.bit`
- `GUI/pynq_multi_fx_gui.py` for RGB frame rendering
- `GUI/audio_lab_gui_bridge.py` for dry-run-first `AppState` to
  `AudioLabOverlay` control mapping

Do not call `run_pynq_hdmi()` for the live AudioLab path. That helper still
belongs to the old PYNQ base-overlay style and would load another full
overlay such as `base.bit`, replacing the AudioLab DSP bitstream.

## Files

| File | Role |
| --- | --- |
| `pynq_multi_fx_gui.py` | Renderer, optional desktop preview helpers, render cache, and 1280x720 / 800x480 frame functions. |
| `audio_lab_gui_bridge.py` | Maps visual `AppState` changes to existing `AudioLabOverlay` APIs. It defaults to dry-run and never loads a bitstream itself. |
| `fx_gui_state.json` | Runtime state generated by preview use; ignored by git. |

The legacy untracked `HDMI/` experiment tree was removed after confirming it
was not used by the current deploy, tests, or HDMI runtime scripts.

## Live HDMI Runtime

The deployed Phase 4/5 HDMI design still emits a fixed `1280x720` HDMI
signal. For the 5-inch 800x480 LCD, Phase 5A output mapping and user visual
inspection showed that the practical visible viewport is the top-left
`800x480` region:

```text
x=0, y=0, w=800, h=480
```

The standard 5-inch LCD check is:

```sh
ssh xilinx@192.168.1.9 '
  cd /home/xilinx/Audio-Lab-PYNQ &&
  sudo env PYTHONPATH=/home/xilinx/Audio-Lab-PYNQ \
    python3 scripts/test_hdmi_800x480_frame.py \
      --variant compact-v2 \
      --placement manual \
      --offset-x 0 \
      --offset-y 0 \
      --hold-seconds 60
'
```

`--placement center` and offset sweeps are kept only for diagnostics. They
are not the default operating mode for this LCD.

## Renderer API

```python
from pynq_multi_fx_gui import AppState, render_frame, render_frame_800x480

state = AppState()
full = render_frame(state)                         # (720, 1280, 3), uint8
small = render_frame_800x480(state, variant="compact-v2")  # (480, 800, 3)
```

The renderer returns RGB `numpy.ndarray` frames. The HDMI backend packs the
RGB888 frame into the deployed DDR framebuffer order expected by the VDMA /
`v_axi4s_vid_out` / `rgb2dvi` path.

## Desktop Preview

`pynq_multi_fx_gui.py` still contains optional Tkinter preview code for local
rendering and layout work. Preview use is separate from the live PYNQ HDMI
runtime:

```sh
python3 GUI/pynq_multi_fx_gui.py --window
python3 GUI/pynq_multi_fx_gui.py --bench
```

The preview may save `GUI/fx_gui_state.json`; that file is runtime state and
is ignored by git.

## Control Bridge

`audio_lab_gui_bridge.py` is intentionally separated from drawing:

- it does not instantiate `AudioLabOverlay`
- it does not load any bitstream
- it defaults to `dry_run=True`
- it writes hardware only when the caller passes an already-loaded overlay
  and explicitly disables dry-run mode

Chain reorder remains display-only for the live hardware. The deployed DSP
pipeline order is fixed and must not be implied to be dynamically
reorderable.
